Panorama for Oracle

„Panorama” is my free tool for performance analysis and troubleshooting of Oracle databases.
Completed over many years it contains solutions for analysis workflows based on practical experience.
As a GUI-application it ensures quick reaction with predefined workflows instead off dealing by hand with lots of SQL scripts.

What does it offer?

  • Support for:
    • Performance-optimization of applications using Oracle databases
    • Analysis of runtimes and wait events, also back in history
    • Preparation of several Oracle database's internal information for easy evaluation
  • it focuses on usage by DBAs and also by software developers with less Oracle knowledge
  • aims to many issues that are inadequately analyzed and presented by other existing tools such as Enterprise Manager
  • Panorama only reads via SELECT-SQLs from database. No write access, structure change, compile etc. needed.
  • Supports:
    • Oracle databases on standard hardware as well as RAC-systems and engineered systems (EXADATA)
    • Oracle database versions beginning with Oracle 10.2

Which preconditions are required?

  • Java runtime environment, version 7 or above to run Panorama application.
    (Some problems may occur with IBM JVM. If so please use JVM from SUN/Oracle.)
  • Parts of Oracle's "Diagnostic Pack" are used for some functions of panorama, so licensing of this option is required for connected Oracle-DB
  • Web browser to access Panorama-GUI
  • Your Oracle-User used for login with Panorama requires the grant "SELECT ANY DICTIONARY"

How to run it?

Start running quickly

  • Download file Panorama.war
  • Type "java -jar Panorama.war" in a terminal window, no matter if your are running Windows, Linux or Mac-OS
  • This starts the embedded Winstone application server within one or two minutes.
  • Now you can reach Panorama's login screen in your browser via "http://<your server ip>:8080/Panorama"

Customize your installation

  • You can specify the following additional Java-options executing "java <options> -jar Panorama.war":
    • "-Xmx1024m" :maximum allowed memory allocation for JVM, minimum value of 1024 MB suggested for processing of large results
    • "XX:MaxPermSize=512M" : object space for java classes, suggested value 512MB
    • "-Djava.io.tmpdir=mydir" : define working directory for exraction of jar file, default is system's temp dir
    • "-Djruby.compile.invokedynamic=true" : Fix possible problems with some JREs resulting in "java.lang.ClassNotFoundException:"
  • You can specify additional Winstone-options executing "java -jar Panorama.war <options>":
    • Allow only SSL encrypted https-connections with disabled http-port:
      java -jar Panorama.war --httpPort=-1 --httpsPort=8443 --httpsKeyStore=<keystore file> --httpsKeyStorePassword=<keystore password>
    • Use another application prefix to access via "http://<your server ip>:8080/MyPrefix": java -jar Panorama.war --prefix=/MyPrefix
    • All command line options of embedded winstone server are described here: http://winstone.sourceforge.net
  • Here you can find a sample shell-script to start Panorama for Unix/Linux
  • Panorama.war can also be run as web archive in any Java JEE application server or web container like Tomcat, Glassfish, JBoss etc. .
  • Panorama writes some encrypted session state information and usage log to local filesystem.
    You can control the directory for this files by setting environment variable PANORAMA_VAR_HOME to your target directory.

What about security?

  • For production purposes you should run Panorama with SSL-encrypted web traffic (https). How to for embedded Winstone application server is described above.
  • Your oracle password is stored encrypted in server memory and will be decrypted shortly only for the process of establishing connection to your target database.
    The key used for this encryption consists of a server key which is salted with a unique client key specific to your browser.
    This way only requests from your browser are able to reuse the stored connect info for reestablishing the database connection.
  • If you retain your login info for further fast logins (checkbox 'Save logins' in statup dialog), this info is stored in local server file system completely encrypted the same way like passwords.
  • You can customize the server key used for encryption by setting the environment variable $SECRET_KEY_BASE to value of your choice.
    Every change of $SECRET_KEY_BASE at server site leads to invalidation of all stored connection info.

Behind the scenes

  • Panorama is an Ruby on Rails application that runs with JRuby. Web resources you can find here:
    • Source code
    • Version history
  • If you use tnsnames.ora for TNS name resolution it should be located at server site in $ORACLE_HOME/network/admin or in a directory targeted by environment variable $TNS_ADMIN.
  • Panorama starts multiple server instances in JRuby (default = 3).
    Each server instance hold it's own Oracle-connection to the last used database and reuses this open connection, if the next requests uses the same database connection info.
    If the next request uses another database, then the current connection is closed and opened with the new connection info.
  • Panorama renders one single web page via Ajax-calls. This way the back button of your browser is not supported to return to former page content.
  • Panorama writes short usage information to plain text file 'panorama_usage_info.log' in the directory targeted by $PANORAMA_VAR_HOME
    You can view this log info via http://<your server ip>:8080/Panorama/usage/info
